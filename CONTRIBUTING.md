# Contributing to LEAP

LEAP is an open methodology for shipping software where **tests are source, schemas are architecture, and code is a disposable artifact generated by agents**. It's experimental by design. We ship breaking changes. We also ship in public — so your feedback, critique, and forks shape what this becomes.

## What contributions are welcome

**High-signal contributions:**
- **SPEC edits** — the current `SPEC.md` codifies our opinions about what belongs in a LEAP project. Some are probably wrong. File an issue or a PR.
- **MANIFESTO edits** — sharper framing, better examples, typo fixes. All welcome.
- **Examples** — working repos that follow LEAP. Small ones, big ones, opinionated ones. We link from the hub.
- **Stunt artifacts** — rewriting a library with LEAP (like [semver-leap](https://github.com/safitudo/semver-leap)) is a strong contribution. Port `ms`, `chalk`, `Day.js`, `lodash` subsets, `uuid`, or anything with a decent test suite. We'll link it here and compare findings.
- **Counter-arguments** — where does LEAP break? Integration code? UI? Ops scripts? Ambiguous domains? Open an issue with your case — we'd rather see the failure modes early than pretend they don't exist.

**Especially wanted right now:**
- **UI / pixel-perfect** — how does the "tests as contract" model transfer when the spec is visual? Start with `PIXEL_PERFECT.md`, tell us where it falls short, propose alternatives.
- **Model-agnostic verification** — the same specs should generate equivalent code across Claude, GPT, Gemini. Has anyone stress-tested this? Results?

## How to contribute

### For small fixes (typos, clarifications)
Fork → PR. We'll review quickly.

### For methodology changes (SPEC, MANIFESTO)
**Open an issue first.** We'd rather align on "is this change worth making?" before you spend effort on a PR. If the issue gets a 👍 from maintainers, open the PR.

### For new examples or stunts
Fork, build, link your repo in an issue titled `[example] <name>`. We'll review and, if it's a good fit, add it to the hub as a linked reference.

### For long-form questions / ideas
Use **Discussions**, not issues. Issues are for actionable work; discussions are for open-ended exploration.

## What we won't merge

- **Private-domain framing.** LEAP is MIT-licensed and open by design. PRs that make it harder to fork, copy, or re-license will be closed.
- **Over-engineering.** If a change adds more abstraction than it pays back, we'll push back.
- **"Add my tool" PRs.** We'll link to legitimately useful tools in examples, not in the core methodology docs.

## Ground rules

- Be kind. We're here to improve the methodology, not to rank each other.
- Cite your evidence. "This breaks for X" with a repo link > "this breaks for X" as an assertion.
- Small PRs > big PRs. Rip changes into reviewable chunks.
